# REST: теория

## Определение

**REST** (Representational State Transfer — передача репрезентативного состояния) — **архитектурный стиль** проектирования распределённых гипермедийных систем. Термин введён Роем Филдингом (Roy Fielding) в его докторской диссертации «Архитектурные стили и проектирование сетевых программных архитектур» (Калифорнийский университет, Ирвин, 2000 г.).

REST описывает набор ограничений (constraints), которым должна удовлетворять система, чтобы обладать желаемыми свойствами: масштабируемостью, надёжностью, простотой, переносимостью и управляемостью.

---

## Протокол, стандарт или архитектурный стиль?

Это частый вопрос на собеседованиях и защитах дипломных работ.

| Понятие | Определение | Является ли REST? |
|---|---|---|
| **Протокол** | Формальный набор правил обмена данными, жёстко определяющий формат сообщений, порядок их передачи и обработки ошибок (например, HTTP, FTP, SMTP) | **Нет** |
| **Стандарт** | Документ, принятый официальной организацией по стандартизации (ISO, W3C, IETF) и обязательный или рекомендуемый к исполнению | **Нет** |
| **Архитектурный стиль** | Совокупность согласованных архитектурных ограничений, абстрагированная от конкретной реализации | **Да** |

**REST не является протоколом**, потому что не определяет формат байтов в сети, заголовки или правила установки соединения. REST не является стандартом, потому что не принят ни одной стандартизирующей организацией. REST — это **архитектурный стиль**: набор принципов, которым может следовать или не следовать любая реализация.

На практике REST почти всегда реализуется поверх протокола **HTTP/HTTPS**, однако теоретически может использоваться с любым протоколом, поддерживающим его ограничения.

## История

| Год | Событие |
|---|---|
| 2000 | Рой Филдинг защищает диссертацию, вводя термин REST |
| 2000–2004 | REST-подход используется при проектировании HTTP/1.1 и URI |
| 2005–2010 | REST вытесняет SOAP как доминирующий стиль веб-API |
| 2015+ | REST остаётся стандартом де-факто; появляются GraphQL и gRPC как альтернативы для отдельных задач |

Сам HTTP разрабатывался с учётом принципов REST — Филдинг был одним из авторов спецификации HTTP/1.1.

## Шесть архитектурных ограничений REST

Филдинг определил шесть ограничений. Система считается RESTful, если она соблюдает первые пять; шестое является необязательным.

---

### 1. Клиент-сервер (Client–Server)

Клиент и сервер разделены и взаимодействуют только через единообразный интерфейс. Клиент не знает ничего о хранении данных на сервере; сервер не знает ничего об интерфейсе пользователя.

**Что это даёт:** независимость развития клиентской и серверной частей, упрощение серверного компонента.

---

### 2. Отсутствие состояния (Stateless)

Каждый запрос от клиента к серверу должен содержать всю информацию, необходимую для его обработки. Сервер не хранит контекст клиентской сессии между запросами.

**Что это даёт:** масштабируемость (любой экземпляр сервера может обработать любой запрос), надёжность (нет риска потери состояния сессии при сбое узла).

**Пример:** аутентификационный токен (JWT) передаётся клиентом в каждом запросе в заголовке `Authorization`, а не хранится в серверной сессии.

---

### 3. Кэшируемость (Cacheable)

Ответы сервера должны явно помечаться как кэшируемые или некэшируемые. Если ответ кэшируемый, клиент или промежуточный узел вправе повторно использовать его для последующих эквивалентных запросов.

**Что это даёт:** снижение нагрузки на сервер, уменьшение задержки для клиента.

**Пример:** HTTP-заголовки `Cache-Control`, `ETag`, `Last-Modified`, `Expires`.

---

### 4. Единообразие интерфейса (Uniform Interface)

Центральное ограничение REST. Все взаимодействия между компонентами системы должны подчиняться единому стандартизированному интерфейсу. Это ограничение включает четыре под-принципа:

#### 4.1 Идентификация ресурсов (Resource Identification)
Каждый ресурс однозначно идентифицируется URI. Ресурс и его представление — разные вещи: один ресурс может иметь несколько представлений (JSON, XML, HTML).

```
GET /articles/42        # ресурс — статья с id=42
GET /users/7/profile    # ресурс — профиль пользователя 7
```

#### 4.2 Манипуляция через представления (Manipulation Through Representations)
Клиент управляет ресурсом, передавая его представление (например, JSON-тело запроса). Если у клиента есть представление ресурса и необходимые права, он может создать, изменить или удалить ресурс.

#### 4.3 Самодостаточность сообщений (Self-Descriptive Messages)
Каждое сообщение содержит достаточно информации для своей обработки: HTTP-метод, заголовок `Content-Type`, код ответа.

#### 4.4 HATEOAS (Hypermedia As The Engine Of Application State)
Клиент взаимодействует с приложением исключительно через гиперссылки, возвращаемые сервером. Клиенту не нужно знать структуру URI заранее. На практике большинство API называют себя RESTful, не реализуя HATEOAS полностью.

```json
{
  "id": 42,
  "title": "Введение в Docker",
  "_links": {
    "self": "/articles/42",
    "author": "/users/7",
    "comments": "/articles/42/comments"
  }
}
```

---

### 5. Многоуровневая система (Layered System)

Клиент не обязан знать, общается ли он напрямую с конечным сервером или через промежуточные узлы (балансировщик нагрузки, API-шлюз, CDN, кэш-прокси). Каждый уровень видит только соседний.

**Что это даёт:** возможность прозрачно добавлять балансировщики, кэши, шлюзы безопасности без изменения клиентского кода.

---

### 6. Код по требованию (Code on Demand) — необязательное

Сервер может расширять функциональность клиента, передавая исполняемый код (JavaScript, апплеты). Единственное необязательное ограничение в REST.

**Пример:** браузер загружает и выполняет JavaScript-бандл — технически это код по требованию.

## Ресурсы и URI

**Ресурс** — любая именуемая абстракция: документ, изображение, коллекция объектов, пользователь, статья.

### Правила проектирования URI

| Правило | Правильно | Неправильно |
|---|---|---|
| Использовать существительные, не глаголы | `/articles` | `/getArticles` |
| Коллекции — во множественном числе | `/users` | `/user` |
| Иерархия через `/` | `/users/7/articles` | `/getUserArticles?id=7` |
| Нижнее подчёркивание или дефис | `/published-articles` | `/publishedArticles` |
| Без завершающего слеша | `/articles/42` | `/articles/42/` |
| Версия API в пути или заголовке | `/api/v1/articles` | — |

## HTTP-методы в контексте REST

| Метод | Операция CRUD | Идемпотентность | Безопасность | Пример |
|---|---|---|---|---|
| **GET** | Read | Да | Да | `GET /articles/42` |
| **POST** | Create | Нет | Нет | `POST /articles` |
| **PUT** | Update (полная замена) | Да | Нет | `PUT /articles/42` |
| **PATCH** | Update (частичное изменение) | Нет* | Нет | `PATCH /articles/42` |
| **DELETE** | Delete | Да | Нет | `DELETE /articles/42` |

> **Идемпотентность** — повторный вызов с теми же параметрами даёт тот же результат, что и однократный. **Безопасность** — метод не изменяет состояние сервера.

---

## HTTP-коды ответов

| Диапазон | Категория | Примеры |
|---|---|---|
| **2xx** | Успех | `200 OK`, `201 Created`, `204 No Content` |
| **3xx** | Перенаправление | `301 Moved Permanently`, `304 Not Modified` |
| **4xx** | Ошибка клиента | `400 Bad Request`, `401 Unauthorized`, `403 Forbidden`, `404 Not Found`, `409 Conflict` |
| **5xx** | Ошибка сервера | `500 Internal Server Error`, `503 Service Unavailable` |

## REST vs SOAP vs GraphQL

| Критерий | REST | SOAP | GraphQL |
|---|---|---|---|
| **Тип** | Архитектурный стиль | Протокол | Язык запросов |
| **Формат данных** | JSON, XML, любой | XML (обязательно) | JSON |
| **Транспорт** | HTTP (обычно) | HTTP, SMTP, TCP | HTTP |
| **Типизация** | Нет встроенной | Строгая (WSDL) | Строгая (Schema) |
| **Избыточность данных** | Возможна (over/under-fetching) | Высокая | Минимальная |
| **Стандарт** | Нет | W3C | Спецификация Facebook/Meta |
| **Применение** | Веб-API общего назначения | Корпоративные интеграции, банки | Гибкие клиентские запросы, мобильные приложения |

## REST API vs RESTful API

Термины часто используются как синонимы, но формально:

- **REST API** — API, заявляющий о соответствии REST.
- **RESTful API** — API, строго следующий всем шести ограничениям Филдинга, включая HATEOAS.

На практике большинство современных «REST API» не реализуют HATEOAS и правильнее называются **«HTTP API»** или **«REST-like API»**. Тем не менее в индустрии оба термина употребляются взаимозаменяемо.

## Форматы представлений

REST не диктует формат данных. Формат согласовывается через механизм **Content Negotiation**:

- Клиент указывает предпочтительный формат: `Accept: application/json`
- Сервер указывает формат ответа: `Content-Type: application/json`

Наиболее распространённые форматы:

| Формат | MIME-тип | Применение |
|---|---|---|
| **JSON** | `application/json` | Де-факто стандарт для веб-API |
| **XML** | `application/xml` | Корпоративные системы, SOAP-совместимость |
| **HTML** | `text/html` | Браузерные клиенты, HATEOAS |
| **Multipart** | `multipart/form-data` | Загрузка файлов |

## Преимущества и недостатки REST

### Преимущества

- **Простота** — использует стандарты HTTP, не требует дополнительных библиотек
- **Масштабируемость** — stateless-архитектура позволяет горизонтально масштабировать серверы
- **Читаемость** — URI интуитивно понятны, JSON легко читается человеком
- **Кэшируемость** — HTTP-кэширование работает «из коробки»
- **Независимость клиента и сервера** — любой клиент, поддерживающий HTTP, может работать с REST API

### Недостатки

- **Over-fetching** — сервер возвращает больше данных, чем нужно клиенту
- **Under-fetching** — для получения связанных данных требуется несколько запросов
- **Отсутствие строгой типизации** — в отличие от GraphQL или gRPC нет встроенной схемы
- **Версионирование** — нет стандартного механизма эволюции API без поломки клиентов
